\chapter{Research Questions and Methodology}
\label{chap:methodology}

This chapter translates the motivation established in Chapter~\ref{chap:background}
into four precise research questions and describes the experimental design
through which each question is answered.
Section~\ref{sec:rqs} states the research questions.
Section~\ref{sec:design} outlines the overall study design.
Sections~\ref{sec:track-a}--\ref{sec:dynamic-method} detail the
two dataset tracks and the static and dynamic evaluation methodologies.
Section~\ref{sec:metrics} defines all evaluation metrics.
Section~\ref{sec:rq-map} maps each research question to the corresponding
dataset, tasks, metrics, and expected outputs.

\section{Research Questions}
\label{sec:rqs}

The central concern of this thesis is how the quality of LLM-generated
graphical user interfaces can be compared with a reusable multi-metric
benchmark and leaderboard. The primary leaderboard ranks generated GUI
submissions, aggregated by generator model and requirement set. Static
visual and requirement-based metrics, dynamic functionality signals, human
review, and LLM-based scoring are used as measurement instruments rather
than as the main objects of comparison. Four research questions
operationalise this concern.

\paragraph{RQ1 -- UIClip-style baseline and pairwise evaluation limits.}
\begin{quote}
\textit{What does a reduced UIClip-style reproduction reveal about
pairwise GUI-quality judgement, and what limitations motivate a shift
toward a generated-GUI leaderboard with richer metrics?}
\end{quote}
This question targets the reproduction aspect of the study
(Track~A, Section~\ref{sec:track-a}).
It uses the UIClip dataset as an anchor for pairwise screenshot
evaluation, but its role is methodological and limited: to establish a
baseline, identify weaknesses of single binary preferences, and motivate
the move toward richer multi-metric evaluation of generated interfaces.

\paragraph{RQ2 -- Static multi-metric quality of LLM-generated GUIs.}
\begin{quote}
\textit{How can generated GUI submissions be compared with static
multi-dimensional metrics across visual structure, information clarity,
and requirement fidelity?}
\end{quote}
This question is centred on Track~B (Section~\ref{sec:track-b}), where
different generator models produce interfaces from the same requirements,
prototype screenshots, and assets. It examines whether the static portion
of the leaderboard can provide interpretable per-dimension quality signals
rather than only a single aggregate score.

\paragraph{RQ3 -- Generation protocol and evaluation feasibility.}
\begin{quote}
\textit{Which generation protocol, validity gates, and scoring components
provide reproducible and cost-effective measurements for a generated-GUI
leaderboard?}
\end{quote}
This question is answered mainly through Track~B and remains a
methodological supporting question. It fixes the leaderboard unit as the
generated HTML submission, requires the same generation prompt version
within a comparison batch, records exact generator metadata, and applies
rendering and static validity gates before scoring. The
LLM-as-a-judge literature~\cite{llm_judge,
li-etal-2025-generation,shi-etal-2025-judging} is used only to motivate
caution when automated scorers are used; judge-model bias is acknowledged
as a limitation rather than studied as the thesis's main object.

\paragraph{RQ4 -- Static and dynamic GUI quality signals.}
\begin{quote}
\textit{For the same set of executable Track~B interfaces,
how are static quality scores related to dynamic functionality and
task-validation outcomes, and where do the two kinds of metrics
complement or diverge from each other?}
\end{quote}
This is the core empirical question of the thesis.
Prior work has evaluated GUI quality from screenshots
or evaluated agents on web tasks, but the two lines of research
have not been connected on the same set of interfaces.
RQ4 addresses this gap by pairing a static rubric score vector
with dynamic workflow-validation measurements for every Track~B item
and analysing both correlations and mismatch cases. The goal is not to
assume that static and dynamic metrics should be identical, but to
understand what each signal captures about the quality of generated GUIs.

\section{Overall Study Design}
\label{sec:design}

Figure~\ref{fig:pipeline} illustrates the end-to-end study design.
The evaluation is organised around two complementary dataset tracks.

\textbf{Track~A} consists of GUI screenshot pairs drawn from the UIClip
dataset.
Its primary function is reproduction and anchoring: by running modern
multimodal LLMs on the same pairwise task that UIClip used, Track~A
enables a direct numerical comparison that establishes whether
contemporary models represent a measurable advance.
Track~A items undergo static evaluation only.

\textbf{Track~B} consists of graphical user interfaces generated from
natural language requirements.
Every Track~B interface is kept in executable form so that it can be
assessed both statically -- through rubric-based scoring -- and
dynamically, through workflow-derived route and content validation
against the rendered HTML.
The key design principle is that the same interface receives both a
static rubric score vector and dynamic workflow-validation measurements,
forming the paired data needed to answer RQ4.

The primary leaderboard unit is the generated GUI submission, aggregated
by generator model and requirement set. Evaluator models, prompts, and
human annotations are part of the evaluation protocol used to compute
the metrics; they are not the main entities being ranked.

Human review provides an independent quality reference for selected
items, but the leaderboard unit remains the generated GUI submission.
Track~A reuses the preference labels published with the UIClip dataset,
so no new annotation is required for pairwise tasks on that track.
Track~B may include fresh human rubric scores for calibration and
qualitative validation, collected according to the annotation protocol
described in the benchmark chapter.

All experiments are documented with exact model identifiers, API version
dates, prompt text, random seeds, and output files to support
reproducibility.

\section{Track A: Screenshot-based Reproduction}
\label{sec:track-a}

Track~A draws on the publicly available UIClip dataset~\cite{uiclip},
which contains pairs of GUI screenshots each labelled with a human
preference indicating which interface is visually superior.
A random sample of 50 to 100 pairs is selected, stratified to cover
the range of quality contrasts present in the original dataset.

No new human annotation is collected for Track~A.
The original UIClip preference labels serve as a lightweight reference
for the reduced reproduction.
This design makes Track~A lightweight and directly comparable to
the UIClip publication while keeping the main thesis contribution --
a static and dynamic leaderboard for LLM-generated GUIs -- concentrated
in Track~B.

\section{Track B: Requirement-driven Generated Interfaces}
\label{sec:track-b}

\subsection{Requirement Sources}

Track~B requirements are collected from two sources.
The primary source is a reduced subset of Vision2Web~\cite{vision2web},
a hierarchical benchmark for visual website development with static,
interactive frontend, and full-stack levels.
This thesis uses only Level~1 static webpage tasks and Level~2
interactive frontend tasks.
Level~3 full-stack tasks are excluded because they introduce backend
state, deployment, and long-horizon engineering requirements that are
outside the scope of a controlled GUI-quality benchmark.

For each selected Vision2Web task, the provided visual prototype,
structured requirement, and available assets are used as the generation
input. Level~1 items support static rubric scoring; Level~2 items also
support dynamic task-validation because they include interaction-oriented
requirements and testable functional expectations.
Approximately 10 to 20 tasks are used for the pilot phase, with a
possible expansion to a small formal subset after rendering and
annotation feasibility have been verified.

Design2Code~\cite{design2code} is retained as a static UI-to-code
reference point and as a backup source for simple webpage reproduction
cases, but it is no longer the primary Track~B data source.
A small set of author-written form-heavy requirements may be added only
if the selected Vision2Web subset lacks enough clear dynamic-validation
cases.

\subsection{Interface Generation}

Each requirement is submitted to two or three generator models,
producing a set of generated HTML implementations per requirement.
For any direct generator comparison, the same generation prompt version,
input screenshots, requirement text, workflow checks, viewport policy,
temperature, and output-budget policy are used wherever the provider
allows it. Generator identity, provider, model identifier, date, prompt
identifier, and key parameters are recorded in metadata. The leaderboard
compares the quality of the generated interfaces; automated scorers and
human review are used only to compute or audit the metrics.

The current Track~B pilot uses \texttt{TB-GEN-v9} as the frozen
generation prompt. Earlier and later prompt variants are treated as
diagnostic smoke-test iterations unless they are explicitly introduced as
separate experimental conditions. This avoids confounding generator
quality with prompt-version changes.

\subsection{Rendering Pipeline}

Generated HTML files are rendered using a headless browser
(Playwright) at a fixed viewport resolution.
For each item, the pipeline saves a PNG screenshot and a serialised
DOM snapshot.
Items whose rendering produces a blank page or a JavaScript console
error that prevents visible content from loading are excluded and
replaced.
The rendered screenshot and the DOM snapshot together constitute
the input to both the static and dynamic evaluation modules.

Before scoring, generated files also pass a static validity gate. The
gate checks complete HTML, implemented route identifiers, defined route
handlers, and clickable controls for workflow-required actions. Exact
visible-label matching is enforced as a hard requirement only when the
source workflow contains an explicitly quoted label. For semantic targets
such as a link related to a named feature, route reachability is the hard
criterion and an exact-label mismatch is recorded as a diagnostic warning
rather than an automatic generation failure.

\section{Static Evaluation Methodology}
\label{sec:static-method}

\subsection{Scoring Instruments}

Static evaluation can combine human rubric scores, deterministic checks,
and LLM-based rubric scoring. These instruments are used to score
generated GUI submissions, not to create a separate evaluator leaderboard.
When LLM-based scorers are used, exact model identifiers, access dates,
prompt identifiers, and outputs are recorded in the reproducibility
appendix.

\subsection{Prompting Conditions}

Two scoring prompt formats may be used.

\textit{Zero-shot} prompting presents the model with the interface
screenshot and a minimal instruction to indicate which of two
interfaces is better (pairwise) or to score the interface on the
rubric dimensions.
This condition establishes a baseline.

\textit{Rubric-guided} prompting supplies the full rubric with
dimension definitions and per-level score anchors, and requires the
model to return a structured JSON response containing one integer score
per dimension and a one-sentence rationale.
This condition tests whether explicit guidance improves alignment with
the rubric definitions and produces more interpretable scoring outputs.

\subsection{Scoring Safeguards}

Two safeguards are applied where they are relevant to the scoring format.

\textit{Order-swapped pairwise comparison} is used only for Track~A
pairwise reproduction or for optional pairwise audits. It repeats each
pairwise judgement with the two interface screenshots presented in the
reverse order~\cite{shi-etal-2025-judging}.
If a model labels interface~A as better when it appears first but
interface~B as better when it appears first, the judgement is
classified as position-biased.
The position-bias rate is computed per scoring model and prompting
condition for audit purposes.

\textit{Repeated sampling} submits selected scoring prompts three times
with a non-zero sampling temperature.
The self-consistency rate measures the proportion of items on which
all three runs produce the same judgement.

\section{Dynamic Evaluation Methodology}
\label{sec:dynamic-method}

\subsection{Approach}

Full computer-use agents introduce two confounded sources of noise:
failures caused by the interface itself and failures caused by the
agent's own planning, perception, or grounding limitations.
Disentangling these sources would add substantial engineering overhead
and would make the pilot harder to interpret.

The current Track~B dynamic pilot therefore uses deterministic
\textit{workflow route simulation}. For each Vision2Web Level~2 item,
the original \texttt{workflow.json} provides click actions and validation
expectations. The evaluator maps workflow click targets to the generated
HTML route controls, simulates the corresponding route transitions, and
then checks whether the destination route contains the expected content
evidence. The report separates route success from content-validation
success so that navigation failures are not conflated with missing or
incorrect destination content.

Action-plan elicitation with an evaluator model remains a possible
extension for cases where no source workflow exists. In that setting, an
evaluator model would produce a JSON action plan from the screenshot,
DOM snapshot, and task description, and a deterministic validator would
check the plan. This thesis treats that model-derived plan approach as
secondary to the workflow-derived evaluator whenever Vision2Web workflow
metadata is available.

\subsection{Task Derivation}

For Vision2Web Level~2 items, dynamic tasks are derived first from the
provided workflow actions and validations. Each workflow case is treated
as an interaction task with an expected route or content outcome. When a
workflow validation depends on visual or CSS state, such as an active
navigation tab, the pilot records this as a limitation of text-only route
simulation and reserves it for browser DOM/CSS or visual validation.

For additional author-written or static-only items, one to two tasks may
be derived from the requirement text. Such tasks must be single
imperative sentences with unambiguous success criteria that can be
checked by inspecting the DOM or a screenshot of the expected end state.

\subsection{Failure Taxonomy}

Dynamic workflow failures are classified into four categories to support
interpretable analysis of RQ4.

\textbf{F1 -- Route failure}: a workflow-required control is absent,
not clickable, or does not reach the expected route.

\textbf{F2 -- Content-validation failure}: the route is reachable, but
the destination content does not contain the expected evidence.

\textbf{F3 -- Validation-model limitation}: the expected evidence
requires browser state, CSS, screenshots, or visual recognition beyond
the current deterministic text/route evaluator.

\textbf{F4 -- Tool or provider failure}: the validation script,
renderer, or generation provider fails before a generated interface can
be evaluated.

Cases in category~F3 and F4 are reviewed manually and excluded from
the RQ4 correlation analysis if they cannot be resolved.

\section{Metrics}
\label{sec:metrics}

\paragraph{Track A agreement (pairwise).}
Cohen's $\kappa$ measures agreement between a pairwise judgement and the
UIClip-derived reference label, correcting for chance. This metric is used
only for the reduced Track~A reproduction and optional scoring audits.

\paragraph{Static rubric score.}
The static leaderboard component is computed from per-dimension rubric
scores. The current working rubric groups static judgement into
functional completeness, layout and hierarchy, and visual quality, with
requirement fidelity considered inside functional completeness for
Track~B. These dimensions are treated as pilot scoring components rather
than a final fixed leaderboard formula. Final score weights and any
composite metric are left to the calibration phase; until then, component
scores are reported separately. Where both human and automated scores are
available, Spearman's $\rho$ is reported as a calibration statistic
rather than as the main leaderboard target.

\paragraph{Position-bias rate.}
The proportion of pairwise items on which the model's preference
reverses between the original and order-swapped presentation. This is
reported for Track~A and optional pairwise audits, not as a central
Track~B result.

\paragraph{Self-consistency rate.}
The proportion of items on which all three repeated-sampling runs
produce the same judgement or score.

\paragraph{Generation validity rate.}
The proportion of generated submissions that pass the static validity
gate: complete HTML, non-truncated script and style blocks, implemented
route identifiers, defined click handlers, and clickable workflow-required
controls. Exact visible-label mismatches for semantic workflow targets
are reported as warnings unless the source workflow specified an explicit
quoted label.

\paragraph{Dynamic workflow success.}
Dynamic success is reported with three related quantities. Route success
is the proportion of workflow cases in which the generated interface
reaches the expected route. Content-validation success is the proportion
of workflow cases in which the reached route contains the expected
content evidence. Task success rate is the proportion of workflow cases
that satisfy both route and content validation. These quantities are
reported separately during the pilot because they diagnose different
failure modes.

\paragraph{Static-dynamic relationship.}
Spearman's $\rho$ between the item-level static rubric score
(aggregate and per-dimension) and the item-level dynamic workflow
success metrics,
computed across all Track~B items. In addition, mismatch cases are
reported qualitatively, because a visually strong interface can still
fail a task and a visually plain interface can still be functionally
successful.

\paragraph{API cost.}
Total token cost in USD per generator and scoring component, computed
from provider pricing at the time of the experiment and used to
characterise the feasibility trade-off addressed in RQ3.
